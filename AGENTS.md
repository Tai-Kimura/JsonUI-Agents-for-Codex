## How Multi-Agent Works in This Project

This project uses Codex CLI's multi-agent feature. Agents are defined in `.codex/config.toml`.

**Key concepts:**
- **Agent roles** are specialized workers (orchestrator, spec, setup, implementation, test, modify, investigate)
- **Skills** (`$skill-name`) are invoked by agents for specific tasks
- **`/agent`** command switches between agent threads
- Codex handles sub-agent spawning, routing, and result consolidation automatically

**User commands:**
- `/agent` — list active agent threads
- `/agent jsonui-orchestrator` — switch to orchestrator
- `/agent jsonui-investigate` — switch to investigate agent (READ-ONLY)
- etc.

## The 4 Invariants

Every task must satisfy all four of these.

1. **`jui build` must pass with zero warnings.** One or more warnings = task not done. Fix the root cause; do not silence.
2. **`jui verify --fail-on-diff` must pass with no drift.** Either the spec or the Layout JSON is wrong — fix one side.
3. **`@generated` files are never edited by hand.** Protocols / Interfaces / ViewModelBase / generated VM, Repository, UseCase signatures are regenerated by `jui build`. To change them, edit the spec.
4. **`$jsonui-localize` must run before a screen is considered done.** `jui build` does not detect hardcoded user-visible strings; localize is the process gate that covers it.

## MCP-first

Agents should call the `jsonui-mcp-server` (the `jui-tools` MCP) for spec reads, layout reads, component lookups, builds, and verification. Bash shell-outs to the `jui` CLI are reserved for the four commands that have no MCP equivalent: `jui g screen`, `jui migrate-layouts`, `jui lint-generated`, `jui g converter`.

Per-agent MCP tool allowlists should be declared in `.codex/config.toml` under each agent's `allowed_tools` (Codex equivalent of the Claude `tools:` frontmatter).

| Common MCP tools | Purpose |
|---|---|
| `mcp__jui-tools__get_project_config` | Read `jui.config.json` |
| `mcp__jui-tools__list_screen_specs`, `list_layouts`, `list_component_specs` | Inventory |
| `mcp__jui-tools__read_spec_file`, `read_layout_file` | Read files |
| `mcp__jui-tools__lookup_component`, `lookup_attribute`, `search_components` | Component/attribute reference |
| `mcp__jui-tools__get_binding_rules`, `get_modifier_order`, `get_platform_mapping` | Platform rules reference |
| `mcp__jui-tools__jui_generate_project`, `jui_build`, `jui_verify` | Generation + build pipeline |
| `mcp__jui-tools__doc_validate_spec`, `doc_validate_component`, `doc_generate_spec`, `doc_generate_html` | Spec validation + doc generation |
| `mcp__jui-tools__doc_init_spec`, `doc_init_component` | Template creation |

## Workflow 1-3: Any JsonUI work

Tell the user: `/agent conductor`

The conductor will:

1. Read repo state via MCP (`jui.config.json`, screen specs, layouts, component specs)
2. Ask 1-2 follow-up questions
3. Tell the user which `/agent` to switch to next

It handles all of:

- **新規に作る／機能追加** — routes to ground (setup) → define (spec) → implement → test, one screen at a time
- **既存を直す** — バグなら debug (READ-ONLY) 先行で spec 起点の原因調査、その結果から define / implement / navigation-* へ。機能改修なら直接 adapt/implement
- **調査だけ** — debug (READ-ONLY) で spec 起点の構造調査

During Phase 2, new agents (`define`, `ground`, `implement`, `navigation-*`, `debug`) don't all exist yet. The conductor maps to existing agents (`jsonui-spec`, `jsonui-setup`, `jsonui-screen-impl`, `jsonui-investigate`, `jsonui-modify`, etc.) behind the scenes.

**Spec-first bug tracing** — when investigating a bug, always start from the spec, not the stack trace. Symptom → spec-section mapping:

| Symptom | Spec section to inspect first |
|---|---|
| UI 表示異常 | `structure.components` → Layout JSON |
| ボタン無反応 | `stateManagement.eventHandlers` + `dataFlow.viewModel.methods` |
| データ未表示/古い | `dataFlow.viewModel.vars` + `dataFlow.repositories/useCases` |
| 表示/非表示切り替え不良 | `stateManagement.displayLogic` |
| API エラー | `dataFlow.apiEndpoints` + `repositories[].methods[].endpoint` |
| 画面遷移不良 | `userActions` / `transitions` + native navigation code |

Always run the three gate commands as diagnostics: `jui verify --detail`, `jui build`, `doc_validate_spec`.

## Design Philosophy

**The specification is the single source of truth for intent and contract. The Layout JSON is the single source of truth for UI structure.**

- **Spec (`docs/screens/json/*.spec.json`)** — screen purpose, state, data flow, public ViewModel contract. Drives Layout JSON skeleton + VM/Repo/UseCase stub generation.
- **Layout JSON (`docs/screens/layouts/*.json`)** — shared cross-platform UI source. Distributed to each platform's `Layouts/` by `jui build`. Edit in the shared directory only.
- **Generated files (`@generated`)** — Protocols / Interfaces / ViewModelBase / method & var signatures. Never hand-edit. Edit spec, then `jui build` regenerates.

### Platform-specific vs runtime attributes

| Key | Resolution | Purpose |
|---|---|---|
| `platform` (dict) | `jui build` time (static) | iOS/Android/Web differences |
| `responsive` | App runtime (dynamic) | Screen size class switching |

Both can coexist on the same node. `platform` is stripped at build time; `responsive` is preserved for the framework.

## Specification rules

1. **Never interpret user input without confirmation.** If input is ambiguous or incomplete: ask clarifying questions, present your interpretation, wait for explicit confirmation.
2. **Always confirm through dialogue.** When there is any room for interpretation: stop, present options, wait.
3. The user knows their requirements better than you do. Incorrect assumptions propagate to all downstream agents and are expensive to fix later.

## File Placement

### Shared Layouts Directory (Single Source of Truth)

Layout JSON lives in `layouts_directory` (default: `docs/screens/layouts/`). **Never** edit per-platform copies in `my-app-ios/`, `my-app-android/`, `my-app-web/` — `jui build` overwrites them.

### Style Files

Place in `Styles/` within `layouts_directory`.

### Resource Files

Place in `Resources/` within `layouts_directory` (`strings.json`, `colors.json`).

### Generated Files (Do Not Edit)

Files carrying `@generated` are regenerated by `jui build` / `jui generate project`. `jui lint-generated` detects hand-edits.

- Layout generation: `*GeneratedView.{swift,kt}`, `*Data.{swift,kt}`, `*Binding.{swift,kt}`
- ViewModel Protocol sync: `*ViewModelProtocol.{swift,kt}`, `*ViewModelBase.ts`, Repository/UseCase protocols
- In-place patches: inheritance補完, Kotlin `override` insertion

To change a generated signature, edit `dataFlow.viewModel.{methods,vars}` or `dataFlow.{repositories,useCases}[].methods` in the spec — not the Protocol or Impl inheritance.

### Tools Directories (Do Not Edit)

- `sjui_tools/`, `kjui_tools/`, `rjui_tools/`, `jui_tools/`

## What you MUST NOT do

1. **Edit `@generated` files by hand.** Edit the spec.
2. **Commit work that produces `jui build` warnings.** Fix them first.
3. **Skip `$jsonui-localize` "just this once".** It's a gate.
4. **Silently fall back to Bash when an MCP call fails.** Surface the failure.
5. **Bypass `/agent jsonui-investigate` for bug fixes in Workflow 2.** The spec-first trace dramatically improves accuracy.
6. **Invent behavior that is not in the spec.** Ask the user, or update the spec first.
